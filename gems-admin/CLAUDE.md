## What This Is

**GEMS Admin Portal** — a React web app serving as the admin, branding configuration, and data integration hub for the GEMS product. Part of three components: this Web Admin, a Mobile App (Android/iOS), and a Backend Service.

MVP target: end of April 2026.

## Commands

```bash
npm run dev          # start dev server (Vite, port 5173)
npm run build        # production build → dist/
npm run preview      # preview the production build locally
npm run lint         # ESLint
npm run test         # Vitest (watch mode)
npm run test -- --run                            # run all tests once (CI)
npm run test -- --run src/path/to/file.test.tsx  # run a single test file
```

Copy `.env.example` to `.env` and fill in values before running.

## Key Architecture Patterns

### Auth + Protected Routes (`src/App.tsx`)
`ProtectedRoutes` reads `isAuthenticated` from `AuthContext`. Unauthenticated users are redirected to `/login`. Token is persisted in `localStorage` under `gems_token`.

### Theming (`src/context/ThemeContext.tsx`)
`applyTheme(tenantTheme)` writes `--primary` and `--secondary` CSS variables directly onto `document.documentElement`. Call this immediately after a successful login once the tenant profile is fetched. All shadcn/ui components inherit these via Tailwind's `@theme inline` block in `index.css`. Each tenant can also supply up to two logo URLs.

The footer must always read **"Powered by GEMS"** — it appears in `AppShell` sidebar footer and the login page.

### API Client (`src/lib/api.ts`)
Thin fetch wrapper. Reads `gems_token` from localStorage for the `Authorization` header. Throws an `Error` with the server's message on non-2xx responses.

### Integration / Mapping Module (`src/pages/integration/`)
Core data pipeline — build in this order:
1. **Upload step**: accept JSON (including SAP EWM OData responses); parse client-side; show field preview
2. **Mapping step**: user maps source columns → target GEMS fields (`orderNumber`, `material`, `batch`, `sourceBin`, `quantity`); save `MappingConfig` per tenant
3. **AI mapping** (Phase 2): POST file headers to backend, receive suggested mapping; also accepts WSDL / API doc as input
4. **Load step**: POST file + mapping to `POST /orders/import`; display job queue from `GET /orders/jobs`

**EPCIS generation is NOT triggered from this module.** The Integration module only loads order data into the system. EPCIS files are generated by the **mobile application** after a warehouse operator performs a picking or packing operation (scan events). Do not add export/EPCIS buttons to the integration UI — that flow belongs entirely to the mobile app.

### Scandit Integration (`src/pages/integration/` or a dedicated `src/scandit/`)
- Use Scandit Web SDK (camera-based barcode scanning in browser)
- License key from `VITE_SCANDIT_LICENSE_KEY`
- Scan events attach to an order; they are the new data added on top of the loaded CSV orders
- Keep Scandit init code isolated in one wrapper component

## Data Model Concepts

| Entity | Notes |
|---|---|
| Tenant | Customer org; owns theme config, integration settings, users |
| Order | Loaded from JSON; fields: `orderNum`, `material`, `batch`, `bin`, `qty` |
| ScanEvent | Created by mobile app scan; attached to an Order |
| MappingConfig | Saved column→field mapping for a tenant's JSON format |
| ImportJob | Tracks the status of a JSON import (pending → processing → done/failed) |

## MVP vs Phase 2

**MVP (→ April 2026):** Theming, customer registration, user management, JSON upload + field mapping, Scandit integration in mobile app, EPCIS generation in mobile app.

**Phase 2:** SSO (SAML/OIDC), AI-based mapping from WSDL/API docs, outbound to arbitrary data structures, iOS support.
